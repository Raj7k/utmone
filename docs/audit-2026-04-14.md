# utm.one Product Audit — 2026-04-14

**Scope:** Core product paths only — auth, onboarding, workspace, link creation, link redirect, dashboard data. Admin, dev tools, marketing pages, and experimental features are **out of scope** for this pass.

**Status:** Read-only audit. No code changes were made. Treat each finding as "needs verification and fix decision" — some may have been partially addressed by later commits.

**How to read this:** Each finding has a severity, location, root cause, customer impact, and suggested fix. Severities are **CRITICAL** (prod-breaking or data breach risk), **HIGH** (hits common paths often), **MEDIUM** (hits under load or edge cases), **LOW** (cleanup / hardening).

---

## P0 — Do these first (CRITICAL)

### 1. Service role key referenced in a frontend file
- **File:** `src/pages/resources/playbooks/HRKatalystReferralPlaybook.tsx:956`
- **What:** `Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!` appears in a React component. `Deno.env` is Edge Function runtime, not browser — so this code would crash in the browser, but the fact that the string is there is a code smell and potentially a build-time leak risk.
- **Impact:** If the service role key is ever injected at build time (e.g., via Vite env replacement), it ends up in the public bundle. Anyone with that key bypasses every RLS policy.
- **Fix:** Remove this reference. Anything needing service-role access must live in an Edge Function.

### 2. CORS allows `*` on every edge function
- **File:** `supabase/functions/_shared/security-headers.ts:10`, `supabase/functions/_shared/cors.ts:2`
- **What:** Both shared CORS helpers hardcode `Access-Control-Allow-Origin: *`. Every edge function inherits this.
- **Impact:** Any website can call your authenticated edge functions with the user's JWT (if they have it in a cookie / localStorage and the user visits a malicious page). Combined with `Allow-Credentials` this becomes exploitable.
- **Fix:** Replace `*` with an allowlist of origins (`https://utm.one`, `https://app.utm.one`, `https://*.lovableproject.com` for previews). Add per-function overrides only where public access is genuinely required.

### 3. Redirect edge function logs full destination URLs
- **File:** `supabase/functions/redirect/index.ts:246`
- **What:** `console.log(\`Redirecting to: ${finalRedirectUrl} (302)\`)` persists into Supabase logs.
- **Impact:** Destination URLs often contain sensitive query params — OAuth codes, email addresses, session tokens, magic-link tokens. Anyone with Supabase project log access (contractors, future team members, compromised credentials) can harvest them.
- **Fix:** Log only `slug` and a boolean `redirected: true`, or a hash of the URL. Never log the full URL body.

### 4. fast-redirect queries by slug without domain filter
- **File:** `supabase/functions/fast-redirect/index.ts:89`
- **What:** Query filters by slug only. The DB has a composite `(domain, path, slug, status)` unique index — a slug-only query can't use it fully.
- **Impact:** On a busy DB with shared slug namespace across domains (e.g., `utm.click/abc` vs `utm.one/abc`), wrong-domain rows may be returned, or a seq scan kicks in. Adds 50–200ms per redirect and risks serving the wrong destination.
- **Fix:** Always `.eq('domain', domain)` and `.eq('path', path)` before `.eq('slug', slug)`. Parse domain/path from the inbound request URL.

## P1 — Do these next (HIGH)

### 5. OAuth tokens stored unencrypted
- **File:** `supabase/migrations/20251122222325*.sql:218-220`
- **What:** `integrations.access_token`, `integrations.refresh_token`, `integrations.config` are plaintext. RLS lets workspace members read them.
- **Impact:** A compromised workspace member (or someone with RLS-bypassing read access via service role in logs) can exfiltrate live OAuth credentials for HubSpot, Salesforce, Google, Slack, etc.
- **Fix:** Use Supabase Vault for token storage, or encrypt with pgsodium at column level. Never return `access_token`/`refresh_token` to the client — instead return `has_token: bool`.

### 6. Webhook secrets stored in plaintext
- **File:** `supabase/migrations/20251122222325*.sql:206`
- **What:** `webhook_subscriptions.secret` is TEXT, readable by any workspace member.
- **Impact:** A malicious member can sign fake callbacks; if the DB is ever dumped, all outgoing webhook signing secrets are compromised.
- **Fix:** Same as above — Vault or pgsodium. Rotate existing secrets after remediation.

### 7. Redirect path blocks on `isUniqueClick` lookup before returning 302
- **File:** `supabase/functions/redirect/index.ts:214`
- **What:** Uniqueness check (24h lookback on `link_clicks`) runs synchronously before the 302 is sent.
- **Impact:** Adds 100–200ms to every click under load; the browser is blocked waiting for the redirect response. Directly hurts TTI for end-users.
- **Fix:** Move the uniqueness check into the async `logClick` path. Return the 302 first, compute/update `is_unique` in the background.

### 8. Dual triggers fire on every `link_clicks` INSERT
- **File:** `supabase/migrations/20251128010124*.sql` (`trigger_set_click_workspace_id`), `supabase/migrations/20251130060032*.sql` (`trigger_set_click_campaign_id`)
- **What:** Two triggers both do `SELECT ... FROM links WHERE id = NEW.link_id`, once per INSERT.
- **Impact:** ~40–80ms extra per click, rising with DB load. At 100 clicks/sec this is sustained CPU burn.
- **Fix:** Denormalize `workspace_id` and `campaign_id` into the click insert itself — compute them in `track-event` / `process-click-queue` before the INSERT. Drop the triggers.

### 9. `fast-redirect` silently drops writes to `sentinel_saves`, `link_bandits`, `link_cache_metadata`
- **File:** `supabase/functions/fast-redirect/index.ts:215, 262`, `supabase/functions/contextual-redirect/index.ts:122, 195, 215`
- **What:** Fire-and-forget inserts with `.then((r) => log(r.error))` — the promise is never awaited, errors never block.
- **Impact:** If the tables don't exist or RLS rejects, features silently fail. Sentinel dashboards show zero saves. Bandit weights never update. Cache-tier decisions are based on stale data.
- **Fix:** Either await these with explicit error handling, or route them through a `click_queue`-style table with a cron/worker that reconciles on failure.

### 10. `create-public-link` picks an arbitrary workspace for public link ownership
- **File:** `supabase/functions/create-public-link/index.ts:150-170`
- **What:** Function picks "the oldest workspace in the table" as the owner of public links.
- **Impact:** Public links are owned by a random customer's workspace. If that workspace is deleted, all public links orphan. If the owner is angry, they can view/edit/delete public links they didn't create.
- **Fix:** Create a dedicated `system_public_workspace` row (marked with a flag). Query by flag, not by `ORDER BY created_at`. Add a migration to create it if missing.

### 11. Link password brute-force: no rate limit on password verification
- **File:** `supabase/functions/verify-link-password/index.ts` (referenced from redirect flow)
- **What:** No attempt counter, no backoff.
- **Impact:** An attacker with a single protected link can try unlimited passwords. Combined with common password lists, this is trivially crackable.
- **Fix:** Track attempts per `(link_id, ip)` in a rate-limit table. Lock after 5 failures for 15 min. Log failed attempts.

### 12. Slug availability check → insert race
- **File:** `src/components/tools/URLShortenerTool.tsx:140-156`, `src/components/link-forge/Step2Shortener.tsx:103-129`
- **What:** Client checks slug availability, then submits. Another user can take the slug in between.
- **Impact:** ~1% of link creations fail with a cryptic "slug already taken" after the user thought it was available.
- **Fix:** Trust the DB unique constraint. On insert failure with code 23505, auto-regenerate the slug and retry once. Only bubble up the error to the user after 2-3 retries.

## P2 — Address under load or when time allows (MEDIUM)

### 13. Onboarding upsert silently drops ICP fields if column missing
- **File:** `src/pages/OnboardingWizard.tsx:237-252`
- **What:** If the full upsert fails with PGRST204 (column not in schema cache), we retry with only `id/email/full_name` — dropping `user_type`, `icp_role`, `primary_use_case`.
- **Impact:** On DBs that haven't yet run migration `20260414000007`, users' role/goal selections are silently lost. The product has no ICP signal for those users.
- **Fix:** Gate the fallback behind a feature-flag or environment check, OR make the migration a hard prerequisite (fail loudly if columns are missing rather than silently stripping data).

### 14. Workspace switch leaves stale `lastWorkspaceRef`
- **File:** `src/contexts/WorkspaceContext.tsx:96-99`, `src/contexts/AppSessionContext.tsx:182-196`
- **What:** After `switchWorkspace()`, if the new workspace fetch fails, the UI keeps rendering the old workspace from `lastWorkspaceRef`. User thinks they switched; they didn't.
- **Impact:** Users create links in the wrong workspace. Data integrity issue.
- **Fix:** Clear `lastWorkspaceRef` at the start of `switchWorkspace`. Show a loading state until the new workspace resolves. If the switch fails, surface a clear error and let the user pick again.

### 15. XSS risk in pixel-js script embedding
- **File:** `supabase/functions/pixel-js/index.ts:100-143`
- **What:** `pixelId` is interpolated directly into a JavaScript template literal. No sanitization.
- **Impact:** If `pixelId` comes from user input anywhere (URL param, request body), an attacker can inject arbitrary JS that runs on every page embedding the pixel.
- **Fix:** Validate that `pixelId` is a UUID (regex match). Reject anything else. Never embed unvalidated user input in generated JS.

### 16. `fast-redirect` reads columns that may not exist on fresh Supabase projects
- **File:** `supabase/functions/fast-redirect/index.ts:86-91`
- **What:** Selects `cache_priority, health_status, sentinel_enabled, sentinel_config`. Supabase silently returns `undefined` for missing columns instead of erroring.
- **Impact:** On Supabase instances that haven't run late-2025 migrations, the function appears to work but sentinel/health features are no-ops. Hard to detect.
- **Fix:** Add a startup schema-version check; or default missing columns in a `SECURITY DEFINER` RPC that returns sane defaults.

### 17. URL validation is client-side only in several flows
- **File:** `src/components/dashboard/WelcomeModal.tsx`, `src/components/tools/URLShortenerTool.tsx`, `src/components/url-shortener/URLShortenerBasic.tsx`
- **What:** Dashboard/authenticated flows rely on `new URL(...)` in the browser. Only `create-public-link` has server-side validation.
- **Impact:** Authenticated users can bypass client validation (curl, Postman) and create links pointing to `file://`, `javascript:`, local IPs, or internal addresses. Can be weaponized for phishing or SSRF.
- **Fix:** Centralize URL validation in a shared helper or Postgres trigger. Block `file:`, `javascript:`, `data:`, localhost, and RFC1918 addresses server-side.

### 18. AppSession fetches `workspace_members` with nested `workspaces(*)` join
- **File:** `src/contexts/AppSessionContext.tsx:186`
- **What:** Join returns full workspace rows per membership. No pagination or column limit.
- **Impact:** For users in 20+ workspaces, the join is big (multi-KB). Blocks initial session load.
- **Fix:** Select only `workspace_id, role` in the join, then fetch workspace details lazily (or in a second parallel query for only the "current" workspace).

### 19. Rate limit on public endpoints is IP-only and spoofable
- **File:** `supabase/functions/create-public-link/index.ts:17-45`
- **What:** `X-Forwarded-For` is trusted without Cloudflare header verification.
- **Impact:** Attacker spoofs `X-Forwarded-For` to bypass the 10-per-hour limit and spam the public link creator for SEO spam, phishing hops, etc.
- **Fix:** Validate `X-Forwarded-For` only if `CF-Connecting-IP` header is present (Cloudflare's trusted IP). Combine IP + UA fingerprint for the rate key.

### 20. Uniqueness check index order may not match query
- **File:** `supabase/functions/redirect/index.ts:57-64`
- **What:** Query filters by `link_id, ip_address, user_agent, clicked_at` but index order may put `clicked_at` first.
- **Impact:** Query runs 50–100ms per click instead of sub-10ms. Compounds with finding #7.
- **Fix:** `CREATE INDEX idx_link_clicks_uniqueness ON link_clicks(link_id, ip_address, user_agent, clicked_at DESC)`. Run `EXPLAIN ANALYZE` to verify.

### 21. No `password_hash` RLS blocking policy
- **File:** No migration drops or restricts `password_hash` column access
- **What:** Authenticated users can `SELECT password_hash FROM links WHERE workspace_id = theirs` and attempt offline crack.
- **Impact:** Workspace-scoped password hash exposure. Weak passwords are crackable offline.
- **Fix:** Add a column-level RLS restriction, or move password hashes to a separate table `link_passwords` with row-level RLS that only the link creator can read.

## P3 — Cleanup / hardening (LOW)

### 22. Signup & Auth pages do their own session check
- **File:** `src/pages/Signup.tsx:86-109`, `src/pages/Auth.tsx:110-151`
- **What:** Both pages call `supabase.auth.getSession()` manually instead of reading from `AppSessionContext`.
- **Impact:** ~300–500ms of redundant network work on first visit.
- **Fix:** Use `useAppSession().isAuthenticated`.

### 23. Dashboard RPC called with `p_user_id: null` when cache is stale
- **File:** `src/hooks/dashboard/useDashboardCore.ts:79-82, 97`
- **What:** `getCachedUserId()` can be empty on first render; RPC is called anyway with `p_user_id: null`.
- **Impact:** First-render onboarding checklist shows inaccurate counts (e.g., `hasInvitedTeam: false` always).
- **Fix:** Gate the RPC behind `isReady && currentWorkspace && userId`.

### 24. Redirect function returns generic "Link not found" for pending links
- **File:** `supabase/functions/redirect/index.ts:116`
- **What:** `.eq('status', 'active')` returns 404 for any non-active link.
- **Impact:** Links in `pending` approval state show as "not found" — confusing for users whose links are awaiting admin approval.
- **Fix:** Query without the status filter, then branch: `404` for missing, `403` with "pending approval" message for pending, `302` for active.

### 25. Edge function error messages leak schema details
- **File:** `supabase/functions/submit-early-access/index.ts:131-132`
- **What:** `error.message` returned directly (e.g., constraint names, table names).
- **Impact:** Helps attackers map your schema.
- **Fix:** Return a sanitized message to client; log the full error server-side.

### 26. Click logs persist raw IP + user agent
- **File:** `supabase/functions/redirect/index.ts:218-231`
- **What:** `ip_address` and `user_agent` are stored without hashing.
- **Impact:** GDPR/CCPA risk. Users can be re-identified by their IP+UA combo months later.
- **Fix:** Store a salted hash (rotate salt monthly), not the raw values. Define a data retention window and purge older entries via a cron job.

### 27. OPTIONS preflight responses are not `no-cache`
- **File:** `supabase/functions/redirect/index.ts:75`, `supabase/functions/fast-redirect/index.ts:34`, `supabase/functions/contextual-redirect/index.ts:114`
- **What:** No `Cache-Control: no-cache` on OPTIONS responses. Browser may cache a stale CORS response.
- **Impact:** Intermittent "CORS blocked" errors after deploys.
- **Fix:** Add `Cache-Control: no-cache` to OPTIONS responses.
